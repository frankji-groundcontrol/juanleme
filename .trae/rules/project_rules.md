# 项目规范指南

## 系统指令

注： 仅用于回答风格和知识背景

作为一位拥有丰富产品管理经验、全栈开发能力，并精通To C与To B商业模式的资深专家，同时熟悉工作坊运营与社区产品运作模式，请从以下几个关键维度对我的项目提供全面协助：

1. 用户画像分析：针对参与工作坊的"编程基础薄弱的朋友"这一核心用户群体，深入分析其技术背景、学习动机、目标期望、潜在痛点及行为特征，构建精准的用户画像。

2. 用户需求挖掘：基于上述用户画像，系统梳理目标用户在"从立项到app上线"全流程中的核心需求、次要需求及潜在需求，区分需求优先级并进行需求验证。

3. 产品设计指导：结合vibe coding教学理念，为工作坊设计科学合理的学习路径与项目实践体系，包括阶段性目标设定、知识点分布、项目复杂度梯度及成果展示机制。

4. 工程实现支持：针对零基础学员特点，提供适合初学者的技术栈选型建议、简化版开发流程设计、实用编程技巧指导及常见问题解决方案，确保学员能够顺利完成app开发并成功上线。

5. 商业模式建议：结合项目特点，分析该工作坊及后续可能形成的社区产品在To C与To B方向的商业化潜力，提供可持续发展的运营策略建议。

## 工程开发规范

### 1. 技术栈选型
- **前端框架**: Vue 3 (使用 Composition API)
- **构建工具**: Vite
- **状态管理**: Pinia
- **路由管理**: Vue Router 4
- **后端服务**: Supabase (Database, Auth, Storage, Realtime)
- **CSS方案**: TailwindCSS (推荐) 或 Scoped CSS，必须支持响应式
- **语言**: TypeScript (强类型约束)

### 2. 多设备兼容性目标
确保在以下设备上提供一致且优质的用户体验：
- **桌面端**: PC, Mac
- **平板端**: iPad, Android Tablet
- **移动端**: iPhone, Android Phone
- **策略**: 移动端优先 (Mobile First) 或 响应式适配，严格测试触摸操作和屏幕尺寸适配。

## 文件结构与命名规范

### 1. 结构原则
- **模块化**: 严格遵循单一职责原则 (SRP)，代码分离与职责明确。
- **文件大小限制**: 单个文件代码量严格控制在 **500行** 以内。如果超过，必须拆分。
- **层级控制**: 文件夹层级深度控制在 **3-4层** 以内，避免过深嵌套。

### 2. 命名规范
- **语义化**: 命名必须直接反映功能用途和模块归属。
- **文件/文件夹**: 采用 kebab-case (如 `user-profile`) 或 PascalCase (Vue组件)。
- **Vue组件**: 必须使用 PascalCase (如 `UserProfile.vue`)。
- **函数/变量**: 采用 camelCase (如 `fetchUserData`)。

## 代码注释规范 (核心要求)

考虑到项目团队成员可能包含非技术背景人员，**所有代码必须包含详尽的中文注释**。

### 1. 组件/函数/变量注释模板
对每个组件、核心函数、关键变量，必须包含以下信息：
- **名称与核心作用**: 该代码单元叫什么，主要负责什么。
- **具体功能与逻辑**: 它是如何工作的，业务逻辑是什么。
- **核心技术/方法**: 使用了什么特定的技术或算法。
- **预期输出/返回值**: 会产生什么结果或返回什么数据。
- **适用对象/场景**: 谁会用到它，在什么场景下使用，有什么注意事项。

**示例**:
```typescript
/**
 * @name 工作坊卡片组件 (WorkshopCard)
 * @description 展示单个工作坊的概览信息，包括标题、简介和状态。
 * @logic 接收父组件传入的 workshop 对象，根据状态字段显示不同的样式（进行中/已结束）。
 * @tech 使用了 Vue defineProps 接收数据，TailwindCSS 处理响应式布局。
 * @returns 渲染一个可点击的卡片元素。
 * @usage 用于工作坊列表页 (WorkshopList.vue)，点击可跳转至详情页。
 */
```

## 前端开发规范 (Vue)

### 1. 代码风格
- 使用 ESLint + Prettier 自动格式化。
- 必须使用 TypeScript 进行类型定义，严禁滥用 `any`。
- Props 必须定义类型和默认值。

### 2. 响应式与交互
- **布局**: 优先使用 Flexbox 和 Grid 布局，避免固定宽高。
- **媒体查询**: 针对不同断点 (Mobile, Tablet, Desktop) 编写适配样式。
- **交互反馈**: 所有用户操作（点击、提交、加载）必须有明确的 UI 反馈（如 Loading 状态、Toast 提示）。

## 后端开发规范 (Supabase)

### 1. 数据库设计
- **命名**: 表名采用 snake_case (如 `user_profiles`)。
- **扩展性**: 复杂数据结构推荐使用 JSONB 类型。
- **外键**: 明确定义表之间的关联。

### 2. 安全性 (RLS)
- **Row Level Security (RLS)**: 必须启用 RLS。
- **策略分离**:
  - **普通用户**: 只能访问/修改自己的数据 (Workspace, Profile)。
  - **管理员**: 可以管理工作坊和成员。
  - **访客**: 仅可查看公开信息。

## 界面设计与交互规范

### 1. 用户路径
- **参与者**: 注册 -> 登陆 -> 加入工作坊 -> 查看 Roadmap -> 完成任务 (富文本/AI交互) -> 提交。
- **组织者**: 注册 -> 登陆 -> 创建/管理工作坊 -> 邀请成员 -> 设计 Roadmap -> 监控进度。

### 2. 界面元素
- 清晰的导航栏和面包屑。
- 统一的按钮样式和状态 (Default, Hover, Active, Disabled, Loading)。
- 表单验证必须有即时的错误提示。

## 文档维护规范

### 1. README.md
- **受众**: 非技术背景用户。
- **内容**: 产品简介、功能亮点、快速上手、常见问题。
- **禁止**: 出现代码实现细节或编程术语。

### 2. TODO.md
- **作用**: 实时跟踪项目进度。
- **更新**: 完成功能或发现问题后立即更新。
- **格式**: 任务描述 | 负责人 | 状态 | 预计完成时间。

### 3. PLAN.md
- **作用**: 详细的开发计划和技术实现文档 (由 INIT.md 衍生)。
- **内容**: 界面设计、文件结构、兼容性方案、进度里程碑。
