# 卷了么立项文档

# 用户画像
## 关键词
1. 工作坊成员：参加Vibe Coding工作坊来学习如何搭建自己的app或者工作流程
2. 时间灵活：有更多时间去和不同的AI工具互动，来获取完成项目的灵感和技能帮助
3. 编程基础：并不一定来自于任何编程背景，但俱备学习热情
4. 学习路径：并没有有详细的学习路径，需要一个工具来进行规划
## Persona
坏胖胖，男，来自中国，目前待业，参加了名为周三下午搓个垃圾出来的工作坊，过去从事物流运输，算是传统行业背景，没有任何编程基础，但如果有明确的引导，可以快速地做一些产品展示。

## 用户数量与角色
本产品仅对参与工作坊的社区成员开放。
1. 参与工作坊的成员：使用总人数不超过500人，周活跃用户不超过50人。
2. 组织工作坊的成员：运营团队不吵过5人。

## 用户需求
### 参与工作坊的成员
1. 用户能在现场能够快速加入对应的工作坊的workspace
2. 用户能够在现场与其他用户快速组队，完成课堂作业
3. 需要能够有一个界面友好的引导工具，帮助用户快速上手应用的灵感，产品设计，立项，demo，上线流程
4. 填写内容可以支持markdown格式，用户可以添加图片，网页等内容

### 组织工作坊的成员
1. 能够建立新的工作坊，设置工作坊的名称，邀请成员加入
2. 能够在工作坊中组织成员，分配任务，协调合作
3. 能够创建引导工具，主要是富文本编辑器，加一个chatbot引导用户丰富他们填写文字


## 用户路径
### 参与工作坊的成员
1. 用户通过注册界面注册，使用邮箱和密码注册，填个用户名
2. 用户通过登陆界面登陆
3. 能够看到已加入的工作坊列表
4. 可以通过邀请码加入新的工作坊
5. 可以编辑个人资料，昵称，头像，个人简介，关键词等
6. 点击进入工作坊，可以查看目前所在的活动流程 roadmap
7. 点击roadmap里面的具体卡片，卡片对应某个流程，进入流程界面
8. 按照提示，在富文本编辑器里完成回答
9. 在回答过程中与接入大模型的chatbot交互，获得书写灵感
10. 提交回答，显示完成
11. 完成最后一步后，所有内容汇总，形成一个整体文档，导出txt, md, 或者pdf

### 组织工作坊的成员
1. 用户通过注册界面注册，使用邮箱和密码注册，填个用户名
2. 用户通过登陆界面登陆
3. 能够看到已加入/已创建的工作坊列表
4. 可以建立工作坊，产生邀请码
5. 可以编辑个人资料，昵称，头像，个人简介，关键词等
6. 可以进入创建的工作坊
7. 可以邀请其他成员管理工作坊，使其成为管理员，可以管理项目组成员和活动流程，但不可以设置其他人为管理员
8. 进入空的或已经有的roadmap，新建或编辑流程
9. 进入流程卡片，流程其实就是个问卷，设置一系列的填空题，选择题（单选，多选，可加选项等）
10. 引导其他用户完成一个个问卷。


# 下一步工作计划

基于产品定义文件 INIT.md，制定一份全面、详细且可执行的前后端工程开发计划，并将其完整记录于 PLAN.md 文档中。该计划必须满足多设备兼容性要求，确保在 PC、Mac、iPad、Android 平板、iPhone 和 Android 手机等主流设备上均能提供一致且优质的用户体验。技术架构明确采用 Vue 作为前端框架，Supabase 作为后端服务解决方案。

计划内容应包含以下核心部分，每部分均需提供详尽的实施细节：

1. 界面设计与交互规范：
   - 系统梳理并列出所有必要的用户界面，明确每个界面的功能定位、核心价值及在整体产品流程中的作用
   - 针对每个界面，为不同用户角色（如管理员、普通用户、访客等）详细描述完整的交互流程，包括入口路径、操作步骤、反馈机制及退出方式
   - 明确各交互操作所使用的具体 UI 元素或组件，包括但不限于按钮、表单、弹窗、导航栏等，并说明其状态变化规则
   - 为每个界面提供清晰的预期效果文字描述，必要时可提供参考示例、线框图链接或功能原型说明

2. 文件结构规划：
   - 设计清晰、模块化的前后端文件组织结构，确保代码分离与职责明确
   - 严格遵循单一职责原则，确保每个文件功能明确，避免创建过大或功能混杂的文件（建议单个文件代码量不超过 500 行）
   - 采用语义化的文件和文件夹命名规范，命名应直接反映其功能用途和模块归属
   - 合理创建文件夹层级结构，深度控制在 3-4 层以内，确保代码逻辑清晰、易于维护和未来扩展

3. 代码注释规范：
   - 考虑到项目团队成员编程基础可能有限，所有代码必须包含详尽的中文注释
   - 对每个组件、函数、变量和关键逻辑节点进行系统性注释说明，内容应至少包括：
     * 该代码单元的名称和核心作用
     * 实现的具体功能和业务逻辑
     * 使用的核心技术或实现方法
     * 预期输出结果或返回值说明
     * 适用对象、使用场景及注意事项
   - 确保注释覆盖代码的每个重要部分，形成完整的代码文档，帮助团队成员快速理解整体架构和实现细节

4. 兼容性实现方案：
   - 详细说明如何通过响应式设计原则、设备适配策略和技术手段实现多设备兼容
   - 列出各设备类型需要特别注意的适配要点和技术实现方法，包括但不限于屏幕尺寸适配、触摸操作优化、性能调优策略
   - 制定兼容性测试标准和方法，明确各设备的最低支持版本和功能降级策略

5. 开发进度与里程碑：
   - 基于功能模块和依赖关系，划分合理的开发阶段和具体任务分配
   - 设置清晰的里程碑节点和可量化的交付物时间计划，包括但不限于需求分析完成、设计稿确认、前端开发完成、后端开发完成、联调完成、测试通过等关键节点
   - 为每个任务分配负责人、预计工时和验收标准，建立进度跟踪机制

6. 技术选型细节：
   - 明确 Vue 框架的具体版本（如 Vue 2.x 或 Vue 3.x）和使用的相关生态工具（如 Vue Router、Vuex/Pinia、Vite 等）及其版本号
   - 详细说明 Supabase 的具体服务配置和使用方案，包括数据库设计、认证策略、存储方案和实时功能实现方式
   - 列出其他必要的第三方库或工具（如 UI 组件库、状态管理工具、表单验证库等），并提供选择理由、版本信息和主要应用场景

7. 测试与验收标准：
   - 为每种目标设备环境制定详细的测试用例，覆盖功能测试、界面测试、兼容性测试和性能测试
   - 明确各功能模块和界面元素的验收标准，包括功能完整性、交互流畅性、视觉一致性和性能指标要求
   - 建立测试环境配置指南和缺陷管理流程

请确保计划内容详尽、具体、可执行，同时兼顾代码的可维护性和未来扩展性，所有技术决策需提供充分的理由和实施路径。

此外，需负责维护以下产品文档，确保其准确性和完整性：

1. README.md：采用用户视角编写，假设目标用户为非技术背景人员，使用通俗易懂的语言详细描述产品功能、使用场景和操作方法。避免使用任何编程术语，适当引用工程中的交互信息作为用户体验描述，例如"完成数据输入后，系统将展示一段炫酷的动画，随后生成您的个性化报告"。文档应包含产品简介、核心功能亮点、快速上手指南、常见问题解答等完整内容。

2. TODO.md：作为项目进度跟踪文档，需实时维护项目开发状态。每次完成一个开发步骤或功能模块后，需立即记录完成内容、实施时间和负责人；如开发过程中出现新的问题、需求变更或技术挑战，也需及时添加至文档中，并记录问题描述、影响范围和解决进展。文档格式应清晰有序，便于团队成员快速了解项目当前状态和待办事项。
